## Current Diagnosis

### 1. Partial HUD updates are effectively one-shot

The spec says live transcript updates should appear while the utterance is still in progress.

The current implementation does not really do that:
- `AVAudioEnginePipeline` emits `captureSnapshots` roughly every 200 ms.
- `SessionController.runCaptureSnapshotLoop()` starts a new `transcribePartial(snapshot)` task for each snapshot and cancels the previous one.
- `WhisperKitTranscriber.transcribePartial(_:)` currently waits for a full `kit.transcribe(...)` call to finish, then returns one final trimmed string for that snapshot.
- Although WhisperKit exposes callback partials inside `makeEngine`, the current `transcribePartial(_:)` path throws them away by calling the engine with `{ _ in }`.

Practical effect:
- if snapshot transcription is slower than the snapshot cadence, most partial tasks are canceled before they yield anything
- the user sees, at best, an occasional late update for a fraction of a second rather than visible streaming text

### 2. `ROGER` / `ABORT` detection is still too brittle

The current resolver behavior is internally consistent but too strict for real use:
- in PTT, command words only become commands when the hotkey path ends in `.pttEnded(mode: .command)`
- in OPEN mode, command words only become commands when `isIsolated(...)` passes all thresholds:
  - `endedBySilence == true`
  - `silenceBefore > 1500ms`
  - `duration < 2000ms`
  - `trailingSilenceAfter >= 800ms`

If any one of those conditions fails, the resolver intentionally falls back to `.inject("ROGER")` or `.inject("ABORT")`.

That matches the user's report that these words still get appended to the terminal.

## Recommended Next Implementation Shape

### Task 15: real streaming partials

Preferred approach:
- stop modeling partials as "snapshot in, final string out"
- let `WhisperKitTranscriber` surface callback partials directly during a decode
- have `SessionController` subscribe to those partial emissions and coalesce them by active utterance sequence

Concretely:
- add an async stream for partial callback text emitted from `WhisperKitTranscriber`
- use snapshot audio only to kick off/refresh a decode, not as the unit of HUD display
- avoid canceling partial work so aggressively that no callback text ever becomes visible
- keep stale partial suppression in the controller so old utterances cannot overwrite current HUD state

### Task 16: reliable command detection

Preferred approach:
- keep PTT command-mode semantics strict: if the user explicitly held command mode, `ROGER` / `ABORT` should always map to command outcomes
- relax OPEN-mode standalone-command detection so naturally spoken command words do not require such brittle silence conditions

Likely options:
- reduce reliance on `silenceBefore` and `trailingSilenceAfter` thresholds for a single-word vocabulary match
- preserve safety by still rejecting partial matches like `ROGER that`, `ABORT mission`, `ABORTED`
- if needed, split OPEN dictation command detection into:
  - strict phrase match
  - looser timing gate for single-word command vocabulary

### Task 17: stronger proof

Current mocked tests prove internal state flow, but they do not prove the user-visible behavior that is failing.

Add proof that would have caught the current mismatch:
- partial callback test proving multiple HUD updates during one utterance
- integration harness proving `ROGER` and `ABORT` actually take the send/clear paths without appending literal text
